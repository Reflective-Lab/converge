# EXP-007: Outside Developer Viability — Fixed Keys & Typed Payloads

**Date:** 2026-04-21
**Status:** superseded on payload shape; fixed-key observations remain useful

> 2026-05-15 update: the experiment below predates the hard
> `FactPayload` contract. `ContextKey` is still the coarse routing namespace,
> but `ProposedFact` no longer carries semantic `String` content in process.
> Payload schema is identified by `(family, version)` and serialized only at
> borders.

**Original hypothesis:** The Converge public surface — 6 canonical crates, 9 fixed ContextKeys, string content, typed IDs, compile-fail safety, and 12 examples — is sufficient for an external developer to build a non-trivial domain application without needing internal access or workarounds.
**Falsification criterion:** If any of the following emerge, the hypothesis fails on that axis:
1. A legitimate domain model that cannot be coherently mapped to the 9 ContextKey variants
2. A correctness bug that the original string-content contract allows at compile time but would be caught by typed content
3. A common pattern (state machine, branching logic, multi-phase) that the API makes needlessly difficult
4. Missing error messages or type-level guidance that leaves the developer guessing

**Method:** Stress-test by attempting to model 3 distinct domain problems against the public contract, recording friction points. Then analyze the 9-variant ContextKey as a design constraint: is it a strength (pit of success) or a ceiling (pit of limitation)?

## What We're Testing

The user has invested heavily in:
- Typed IDs (no semantic strings in APIs)
- Separation of concern (pack/kernel/model/provider boundaries)
- Compile-fail tests (7 tests proving you can't bypass the kernel)
- 12 working examples showing intended usage
- Idiomatic Rust (`#[must_use]`, private fields, feature gates)

The remaining open question: is the *abstraction itself* — the convergence loop over 9 semantic slots — general enough for outside adoption?

## The Contract Surface (as an outsider sees it)

```rust
// The ENTIRE domain vocabulary:
enum ContextKey {
    Seeds,        // Initial inputs
    Hypotheses,   // Ideas from analysis
    Strategies,   // Action plans
    Constraints,  // Rules and boundaries
    Signals,      // Environmental observations
    Competitors,  // Competitive intelligence
    Evaluations,  // Assessments and ratings
    Proposals,    // Awaiting validation
    Diagnostic,   // Errors, never blocks convergence
}

// The payload, after the 2026-05-15 typed-payload change:
struct ProposedFact {
    pub key: ContextKey,        // Which slot
    pub id: ProposalId,         // Typed ID (good)
    payload: Arc<dyn ErasedFactPayload>,
    confidence: UnitInterval,
    pub provenance: Provenance,
}
```

## Stress Test 1: Supply Chain Optimization

**Domain concepts:** Suppliers, routes, warehouses, lead times, costs, demand forecasts, SLA compliance, order batches, transportation modes, customs requirements.

**Mapping attempt:**
| Domain Concept | ContextKey | Works? |
|---|---|---|
| Demand forecast → | Seeds | ✓ natural |
| Supplier profiles → | Signals | ✓ observations about the world |
| Route feasibility analysis → | Evaluations | ✓ scoring routes |
| Cost optimization proposals → | Strategies | ✓ action plans |
| SLA hard constraints → | Constraints | ✓ boundaries |
| Compliance checks → | Constraints | ⚠ collision with SLA constraints |
| Order recommendations → | Proposals | ✓ await validation |
| Transportation mode selection → | Strategies | ⚠ collision with cost optimization |
| What-if scenario hypotheses → | Hypotheses | ✓ analysis output |
| Competitive lead time data → | Competitors | ✓ natural fit |

**Friction points:**
- Strategy collision: cost optimization AND transportation mode both want `Strategies`. The `id` field disambiguates, but the outsider has no type-level separation.
- Constraint collision: SLA bounds AND compliance both target `Constraints`. Same mitigation: id-based, not type-based.
- The content of a "route evaluation" vs a "supplier evaluation" is semantically different but both are `String` in `Evaluations`.
- **Verdict: WORKS** — the 9-key model maps supply chain adequately. The id namespace carries the domain specificity. But the outsider must invent their own id conventions, and the compiler won't catch cross-domain confusion.

## Stress Test 2: Multi-Tenant SaaS Provisioning

**Domain concepts:** Tenant onboarding, resource quotas, billing tiers, feature flags, DNS provisioning, database creation, secrets management, health probes, rollback decisions.

**Mapping attempt:**
| Domain Concept | ContextKey | Works? |
|---|---|---|
| Onboarding request → | Seeds | ✓ |
| Environment discovery → | Signals | ✓ |
| Resource quota analysis → | Evaluations | ✓ |
| Provisioning strategy → | Strategies | ✓ |
| Billing tier constraints → | Constraints | ✓ |
| Feature flag decisions → | Proposals | ⚠ these aren't really "proposals" |
| DNS provisioning result → | ? | ❌ This is a SIDE EFFECT, not a fact |
| Database creation result → | ? | ❌ Same: side effect |
| Health probe results → | Signals | ✓ (cycle 2+) |
| Rollback decision → | Proposals | ✓ |

**Friction points:**
- **Side effects don't fit.** The convergence model is about reasoning to a decision — not executing side effects. DNS provisioning is an action, not a fact. The outsider must separate "convergence to decide" from "execution of the decision." The model doesn't guide them here.
- Feature flags as Proposals feels forced — they're configuration, not proposals awaiting human validation.
- **Verdict: PARTIALLY WORKS** — the convergence loop handles the decision-making phase well, but the "act on the decision" phase is outside the model. The outsider needs to understand that Converge decides; something else executes. This is not obvious from the API or examples.

## Stress Test 3: Clinical Trial Protocol Matching

**Domain concepts:** Patient eligibility criteria, inclusion/exclusion rules, biomarker matching, site capacity, regulatory constraints, randomization arms, adverse event monitoring, protocol amendments, consent tracking.

**Mapping attempt:**
| Domain Concept | ContextKey | Works? |
|---|---|---|
| Patient profile → | Seeds | ✓ |
| Biomarker results → | Signals | ✓ |
| Eligibility scoring → | Evaluations | ✓ |
| Exclusion criteria → | Constraints | ✓ |
| Regulatory requirements → | Constraints | ⚠ collision again |
| Protocol match recommendations → | Strategies | ✓ |
| Randomization arm assignment → | Proposals | ✓ (HITL approval) |
| Adverse event signals → | Signals | ⚠ collision with biomarkers |
| Site capacity data → | Competitors | ❓ stretching semantics |
| Consent status → | ? | ❌ Not a fact about reasoning — it's prerequisite state |

**Friction points:**
- Site capacity is NOT "competitive intelligence" — it's resource availability. The outsider is forced to shoehorn or use `Signals`.
- Consent status is prerequisite state that gates whether the convergence should even run. The model has no concept of "pre-conditions" — only Seeds.
- Multiple collision in Constraints (exclusion criteria + regulatory), multiple collision in Signals (biomarkers + adverse events).
- **Verdict: WORKS WITH SEMANTIC STRETCH** — but the outsider will feel that `Competitors` is misnamed for their domain, and "pre-conditions" are missing from the vocabulary.

## Analysis: The Two Pressure Points

### Pressure Point 1: Fixed ContextKey (9 variants)

**Strength:**
- Forces domain models into a convergence-native vocabulary
- Makes suggestors composable across domains (a policy gate doesn't care what domain the Strategies came from)
- Makes the engine generic — no per-domain type parameters
- The compile-fail test proves you can't accidentally invent new keys

**Limitation:**
- 9 keys means domain concepts COLLIDE within a key, distinguished only by `ProposedFact.id` (a String-based ProposalId)
- Some keys don't map well to all domains (`Competitors` is business-centric; clinical trials don't have "competitors")
- There's no semantic grouping WITHIN a key — all Evaluations are a flat list
- The outsider must invent id conventions (`"price:vendor-a"`, `"risk:vendor-a"`) without compiler support

**Is this a ceiling?** Not for the convergence loop itself. The 9 keys are a semantic protocol — like HTTP methods. You don't need `PURCHASE` and `SUBSCRIBE` as separate methods because `POST` with different bodies covers both. Similarly, `Strategies` covers "cost optimization" and "transportation mode" through content differentiation.

This was the useful failure found by the experiment. Converge now has a schema
layer: `FactPayload::FAMILY`, `FactPayload::VERSION`, and border-side
`PayloadRegistry` decoding.

### Pressure Point 2: Typed Payload Discipline

**What we gave up:**
- No compile-time guarantee that a `ContextKey::Evaluations` fact contains valid evaluation JSON
- No way to express "this suggestor reads Evaluations where criterion == 'price'" in the type system
- Suggestors that read wrong content patterns will fail at runtime, not compile time
- Refactoring a content schema requires grep-and-hope, not compiler errors

**What we gained:**
- Maximum flexibility — any domain can use the same engine
- No generic type parameters on Engine, Suggestor, or Context
- Suggestors from different domains compose without type alignment
- Serde boundaries are explicit and local to each suggestor

**The real question:** Does the outsider need compile-time content safety MORE than they need cross-domain composability?

For outsiders building ONE domain application: they'd prefer typed content. Their evaluations all have the same schema, and a typo in a JSON field name costs debugging time.

For outsiders composing ACROSS domains (the Organism use case): untyped content is correct. A policy gate shouldn't need to know the schema of what it's gating.

## Preliminary Findings

The API is well-sealed and idiomatically Rust. The compile-fail tests are excellent — they prove the kernel boundary. The examples show clear patterns.

**What's missing for the outsider is not code — it's conceptual guidance:**

1. **"Converge decides, something else executes"** — not stated clearly anywhere. The SaaS provisioning case reveals this gap. An outsider might try to model side effects as facts.

2. **"ContextKey is a semantic protocol, not a domain vocabulary"** — the outsider needs to understand they're mapping TO convergence semantics, not encoding their domain directly. Like mapping REST resources to HTTP methods.

3. **"Use id conventions for sub-key namespacing"** — the examples show this (`"price:vendor-a"`) but it's not called out as a deliberate pattern.

4. **Confidence is bounded [0.0, 1.0] but not enforced** — the first time an outsider passes 1.5 and the engine silently clamps or rejects, they'll wonder why the type didn't prevent it.

## Outcome

**Confirmed (partially falsified, then fixed).**

The convergence abstraction maps to real domains and the kernel boundary holds. Two gap classes were identified and resolved:

**Conceptual gaps — fixed in `examples/README.md`:**
Three missing frames now documented: "Converge reasons; something else acts", "ContextKey is a semantic protocol", and "Use structured ids for sub-key namespacing."

**Type-level gap — fixed in `converge-pack`:**
`ProposedFact.confidence` was a public `f64` field with no enforcement of the [0.0, 1.0] range. An external developer could pass NaN, infinity, or -0.1 and the type system would not object. Fixed by making the field private with a clamped `with_confidence(f64) -> Self` builder that normalizes non-finite values to 0.0 and clamps the rest. Engine-level runtime rejection of bad confidence is now superseded by construction-time enforcement. Three engine-level negative tests removed; four unit tests added to `fact.rs` proving the clamping behavior.

One accepted limitation remains: `ProposedFact.content` is `String`. This is the correct trade-off — typed content would break cross-domain composability, which is a core property of the engine.
