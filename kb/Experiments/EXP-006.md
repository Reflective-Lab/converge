# EXP-006: Abstraction Sufficiency — Downstream Migration Readiness

**Date:** 2026-04-21
**Status:** confirmed
**Hypothesis:** The 6 canonical Converge contracts (`converge-pack`, `converge-kernel`, `converge-model`, `converge-provider-api`, `converge-protocol`, `converge-client`) are sufficient for all downstream consumers — Organism, Axiom, Helms, Wolfgang, and Hackathon — to express their domain logic without reaching into internal crates or duplicating abstractions.
**Falsification criterion:** Any downstream repo that imports an internal crate (`converge-core`, `converge-provider`, `converge-runtime`, `converge-domain`, `converge-policy`, `converge-analytics`, `converge-knowledge`, `converge-optimization`, `converge-experience`) for something that should be expressible through the public surface falsifies the hypothesis.
**Method:** Dry-run migration audit — map every converge import in each downstream repo to public vs internal, surface gaps, classify by severity.

## Setup

Five downstream repos audited (all pre-migration state, none have adopted v3.x contracts yet):

- `~/dev/work/organism` — rev 40dc92f (git)
- `~/dev/work/axiom` — tag v3.4.0
- `~/dev/work/helms` — tag v3.4.0
- `~/dev/work/wolfgang` — path deps (active co-development)
- `~/dev/work/hackathon` — tag v3.4.0

## Run

Static analysis: grep for `converge_core::`, `converge_provider::`, `converge_runtime::`, `converge_domain::`, `converge_policy::`, `converge_analytics::`, `converge_knowledge::`, `converge_optimization::`, `converge_experience::` across all Rust source files in each repo.

## Evidence

### Per-Repo Findings

| Repo | Internal Deps | Internal Import Sites | Readiness |
|------|--------------|----------------------|-----------|
| Organism | None | 0 (OCR comments only) | GREEN |
| Axiom | converge-provider | 2 | YELLOW |
| Hackathon | converge-domain, converge-policy, converge-provider | 4 | YELLOW |

### Missing Public APIs (by call site count)

**1. Provider selection logic** — affects Axiom, Hackathon, Wolfgang (3 repos)
```
converge_provider::{ChatBackendSelectionConfig, select_healthy_chat_backend}
```
Not in `converge-provider-api`. The selection implementation lives in the internal provider crate but the config and entry-point belong in the public capability contract.

**2. HTTP authentication middleware** — affects Wolfgang (5 HTTP handler files)
```
converge_runtime::http_auth::{AuthenticatedUser, require_auth}
```
No public contract for this. It's a transport concern (not a domain concern) but Wolfgang has no alternative — it must embed the runtime's auth layer.

**3. Core runtime primitives** — affects Helms (9+ files)
```
converge_core::{StopReason, CriterionResult, Context, ContextKey, ContextView, ProposedFact}
```
`StopReason` and `CriterionResult` appear in every Helms truth_runtime module. These read like they belong in `converge-kernel` or `converge-model`, not buried in `converge-core`.

**4. Storage and experience primitives** — affects Helms
```
converge_core::experience_store::{EventQuery, ExperienceEventEnvelope}
converge_core::integrity::TrackedContext
converge_core::traits::{ContextStore, StoreError}
```
Storage abstractions that Helms uses to build its truth-runtime persistence layer. Currently only accessible via `converge-core` internals.

**5. Analytics, optimization, knowledge** — affects Helms
```
converge_analytics::batch::*, converge_analytics::engine::FeatureVector, converge_analytics::model::{ModelConfig, run_batch_inference}
converge_optimization::gate::ObjectiveSpec, converge_optimization::packs::lead_routing::*
converge_knowledge::{KnowledgeBase, KnowledgeEntry, SearchOptions}
```
Helms has built production-grade scoring, lead routing, and knowledge retrieval on these. No public facade exists for any of them.

**6. Domain agents and policy** — affects Helms and Hackathon
```
converge_domain::packs::{ReconciliationMatcherAgent, OverdueDetectorAgent, trust::*}
converge_policy::*
```
Helms drives reconciliation and payment flows through domain agents. Hackathon drives governance through policy evaluation. Neither has a public surface to consume.

### Organism Is the Proof-of-Concept

Organism already uses only public contracts (`converge-pack`, `converge-kernel`, `converge-provider-api`). It is evidence that the kernel boundary works. But Organism is a planning domain — it does not need analytics, knowledge, optimization, or policy. The other repos reveal what the public surface is missing for application-layer consumers.

## Outcome

**Falsified** — the 6 canonical contracts are necessary but not sufficient.

The kernel boundary is real and working (Organism proves it). But the public surface does not cover:

1. **Provider selection** — `ChatBackendSelectionConfig` / `select_healthy_chat_backend` must move to `converge-provider-api` or a thin new crate.
2. **Kernel-adjacent runtime types** — `StopReason`, `CriterionResult` belong in `converge-kernel` or `converge-model`, not `converge-core`.
3. **Storage traits** — `ContextStore`, `StoreError`, `EventQuery` need a home in the public surface or a stable internal API with explicit guarantees.
4. **Analytics / optimization / knowledge facades** — these three internal crates serve real production use cases (Helms). They either need public pack facades or must be officially acknowledged as stable internal APIs with versioned contracts.
5. **Policy evaluation** — governance workloads (Hackathon) need `converge-policy` accessible without importing all of `converge-core`.
6. **HTTP auth** — Wolfgang's coupling to `converge-runtime::http_auth` is a transport concern that should probably stay internal, but Wolfgang needs an official path (e.g., a separate `converge-runtime-axum` crate with explicit status).

**What we learned:** The architecture is sound at the kernel level. The missing layer is a set of *capability facades* that give application-layer consumers (Helms, Wolfgang, Hackathon) a stable, typed path to analytics, optimization, knowledge, policy, and provider selection — without importing the full internal implementation graph. The v3.5 milestone (capability contract realignment) targets part of this, but Helms's breadth of internal usage suggests v3.6 (surface reduction) needs to be scoped to include explicit facade work, not just marking crates `publish = false`.
