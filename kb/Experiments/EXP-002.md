# EXP-002: Context ID Validation and Typed-ID Design

**Hypothesis:** Context IDs should be validated to enforce a consistent format (lowercase alphanumeric + colon/dash, max 128 chars) rather than accepting unbounded, unvalidated strings. Typed-ID newtype will improve type safety and catch errors at compile time.

**Rationale:** Currently, the system accepts any string as an ID, including empty strings, null bytes, and multi-kilobyte values. This creates:
- **Type safety gaps**: IDs passed to different functions could be accidentally malformed
- **Security issues**: Null bytes in IDs could cause injection vulnerabilities in serialization
- **Escaping issues**: Newlines or quotes in IDs could break logging, CSV export, or string interpolation
- **Resource waste**: No limit on ID length allows unbounded memory growth

Validating IDs and wrapping in a `FactId` newtype prevents all of these.

## Validation Constraints

IDs should match the pattern: `[a-z][a-z0-9:_-]{0,127}`

**Character Set:**
- Start: lowercase letter (a-z)
- Continuation: lowercase alphanumeric (a-z, 0-9), colon (:), underscore (_), or dash (-)
- **Why**: machine-readable, git-compatible, URL-safe

**Max Length:** 128 characters
- Enough for descriptive names
- Protects against unbounded allocation
- Aligns with DNS label length limits

## Current Validation Gaps (Evidence)

### Gap 1: Empty ID Acceptance
```rust
ctx.add_input(key, "", "content")  // Currently: Ok(true)
                                   // Should: Err(ValidationError)
```
**Impact**: No minimum length enforcement

### Gap 2: Null Byte Acceptance
```rust
ctx.add_input(key, "id\0injection", "content")  // Currently: Ok(true)
                                                 // Should: Err(ValidationError)
```
**Impact**: Serialization vulnerability (C string termination confusion)

### Gap 3: Newline Acceptance
```rust
ctx.add_input(key, "id\nescaping", "content")  // Currently: Ok(true)
                                               // Should: Err(ValidationError)
```
**Impact**: Breaks CSV, JSON, and log escaping

### Gap 4: Unbounded Length
```rust
ctx.add_input(key, "a".repeat(10_000), "content")  // Currently: Ok(true)
                                                    // Should: Err(ValidationError)
```
**Impact**: No max-length enforcement, potential DoS

### Gap 5: Whitespace-Only ID
```rust
ctx.add_input(key, "   ", "content")  // Currently: Ok(true)
                                      // Should: Err(ValidationError)
```
**Impact**: Ambiguous in display and hard to debug

### Gap 6: Uppercase Letters
```rust
ctx.add_input(key, "MyID", "content")  // Currently: Ok(true)
                                       // Should: Err(ValidationError)
```
**Impact**: Inconsistent casing complicates deduplication and searching

## Proposed ADR: Typed-ID Newtype

### Before (Current)
```rust
pub trait Context {
    fn add_input(&mut self, key: ContextKey, id: &str, content: &str) -> Result<bool>;
}
```

### After (Proposed)
```rust
#[derive(Clone, Debug, PartialEq, Eq, Hash)]
pub struct FactId(String);

impl FactId {
    /// Parse and validate ID string.
    pub fn new(id: &str) -> Result<Self, IdError> {
        if id.is_empty() {
            return Err(IdError::Empty);
        }
        if id.len() > 128 {
            return Err(IdError::TooLong(id.len()));
        }
        if !id.chars().next().map_or(false, |c| c.is_ascii_lowercase()) {
            return Err(IdError::InvalidStart);
        }
        if !id[1..].chars().all(|c| {
            c.is_ascii_lowercase() || c.is_ascii_digit() || c == ':' || c == '_' || c == '-'
        }) {
            return Err(IdError::InvalidCharacters);
        }
        Ok(FactId(id.to_string()))
    }
}

pub trait Context {
    fn add_input(&mut self, key: ContextKey, id: FactId, content: &str) -> Result<bool>;
}
```

### Benefits

1. **Compile-time assurance**: ID type guarantees validity
2. **Impossible states eliminated**: No way to construct invalid FactId
3. **Better error messages**: `IdError::TooLong(256)` is clearer than generic validation error
4. **Performance**: No runtime validation in hot path (validation only at boundary)
5. **Documentation**: Type signature makes constraints explicit
