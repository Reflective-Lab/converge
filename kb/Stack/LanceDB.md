# LanceDB

LanceDB serves two roles: experience store and vector search engine. The v3.8
boundary moves the concrete LanceDB adapters out of the foundation and into
extensions.

## Experience Store

Implements the `ExperienceStore` [[Architecture/Ports|port]] with vector-indexed event retrieval.

```rust
LanceDbConfig {
    uri: "/data/converge-lance",
    embedding_dim: 384,
}
```

### Vector Event Types

| Type | Purpose |
|---|---|
| `VectorEvent` | Event with embedding vector |
| `SimilarEvent` | Search result with cosine distance |
| `ExecutableQuery` | Vector similarity query |

Agents can find *semantically similar* past events, not just exact matches. "What happened last time we evaluated a vendor like this?"

## Vector Search

Implements the `VectorRecall` [[Architecture/Ports|port]]. Used for similarity search over embeddings generated by the `Embedding` port (FastEmbed, Ollama, cloud APIs).

## Storage Format

Arrow columnar format (via `arrow-schema`, `arrow-array`). RecordBatch-based. Efficient for analytical queries over event history.

## Why LanceDB

- Embedded (no server required) or remote
- Native vector search without a separate service
- Arrow-native — interops with Polars
- Append-optimized — fits the append-only truth model ([[Philosophy/Nine Axioms#3. Append-Only Truth|Axiom 3]])

## v3.8 Boundary

LanceDB belongs with the capability it serves:

- Knowledge recall and memory live in `mnemos`.
- Generic vector search can live in `manifold` if it is exposed as an
  interchangeable capability.
- Runtime persistence is wired by the host runtime.
